== Purpose

This document captures the architecture, component design, data flow, and key technical decisions for the Physical AI Podman Desktop extension. It is a living document вҖ” updated as the project evolves.

For work breakdown and progress tracking, see the link:podman-extension-plan.md[plan doc].

== Goals

[cols="1,3"]
|===
| Goal | Description

| Zero-CLI robotics dev
| Robotics engineers should be able to go from "install extension" to "running simulation" without touching a terminal

| Container-first robotics
| Provide curated ROS2 base images (currently Ubuntu 24.04) with a migration path toward Fedora/RHEL as official ROS2 packaging matures

| Local-to-cluster path
| Enable the transition from local Podman development to OpenShift deployment

| ROSCon demo-ready
| MVP must be demonstrable at ROSCon Toronto, September 2026
|===

== Component Design

=== Extension Backend (`packages/backend`)

The backend is the entry point loaded by Podman Desktop. It manages the extension lifecycle and provides the API surface for the frontend.

==== Responsibilities

* Extension activation/deactivation (`activate()` / `deactivate()`)
* Webview panel creation and media asset loading
* RPC bridge registration via `RpcExtension`
* Command registration (`physical-ai.open` вҖ” reopens the dashboard panel)
* Podman Engine interaction (image build/push/pull, container/pod lifecycle)
* Image build orchestration with `AbortController`-based cancellation
* Build/push progress tracking via in-memory `Map` instances
* Quay.io registry API queries

==== Key Files

[cols="1,2"]
|===
| File | Role

| `src/extension.ts` | Entry point вҖ” creates webview panel, wires RPC, registers `physical-ai.open` command
| `src/api-impl.ts` | `PhysicalAiApiImpl` вҖ” implements all 15 API methods; private helpers use JS `#private` syntax (not RPC-callable)
| `src/progressLogs.ts` | `appendProgressLog()` вҖ” capped at 500 lines with truncation marker
| `package.json` | Extension manifest (contributes: commands, configuration with 6 properties)
| `vite.config.js` | Builds to CJS, externalizes `@podman-desktop/api`
| `assets/ros2-jazzy-base/` | Bundled Containerfile + entrypoint for base image builds
| `assets/ros2-humble-turtlebot3/` | Bundled Containerfile + entrypoint for simulation image builds
|===

=== Extension Frontend (`packages/frontend`)

The frontend is a Svelte 5 SPA rendered inside a Podman Desktop webview panel. It uses TailwindCSS for layout and `.pai-*` CSS classes (defined in `app.css`) for theming that adapts to Podman Desktop's light/dark mode.

==== Responsibilities

* Dashboard landing page with feature cards
* Image catalog browser with pull progress _(APPENG-5770, done)_
* Build & Push for base and simulation images _(done)_
* Simulation Setup wizard _(done вҖ” config + build; launch is Story 2)_
* In-extension Help page _(done)_
* ROS2 topic monitor _(APPENG-5773, planned)_
* Fleet status dashboard _(APPENG-5776, stretch)_

==== Key Files

[cols="1,2"]
|===
| File | Role

| `src/App.svelte` | Root component, hash-based routing via tinro
| `src/Dashboard.svelte` | Landing page with quick-link cards
| `src/ImageCatalog.svelte` | Browse Quay repos, pull images, show local images
| `src/BuildBaseImage.svelte` | Build & push the ROS2 Jazzy base image
| `src/SimulationSetup.svelte` | Wizard for robot/distro/middleware/engine selection + build & push simulation image
| `src/Help.svelte` | In-extension documentation
| `src/lib/BuildPushPanel.svelte` | Shared component for build/push/cancel lifecycle with polling (used by BuildBaseImage and SimulationSetup)
| `src/app.css` | `.pai-*` theme classes (buttons, links, text colors, cards, progress bars) using `--pd-*` CSS variables
| `src/api/client.ts` | RPC client вҖ” creates proxy to backend API
| `src/lib/Route.svelte` | Route component with state persistence
|===

=== Shared Layer (`packages/shared`)

The shared package defines the API contract between frontend and backend, provides the RPC transport, and contains shared type definitions.

==== MessageProxy RPC Bridge

[source]
----
Frontend (RpcBrowser)                    Backend (RpcExtension)
        в”Ӯ                                        в”Ӯ
        в”Ӯв”Җв”Җв”Җв”Җ IMessageRequest в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–әв”Ӯ
        в”Ӯ     { id, channel, args }              в”Ӯ
        в”Ӯ                                        в”Ӯв”Җв”Җ calls registered method
        в”Ӯв—„в”Җв”Җв”Җ IMessageResponse в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ӯ
        в”Ӯ     { id, channel, body, status }      в”Ӯ
----

* Frontend calls `physicalAiClient.someMethod(args)` вҖ” the proxy serializes this as an `IMessageRequest` via `postMessage`
* Backend `RpcExtension` dispatches to the registered method on `PhysicalAiApiImpl`
* Response is sent back as `IMessageResponse` with status `success` or `error`
* 10-second timeout per request
* Podman Desktop webview does NOT deliver unsolicited backend-to-frontend messages вҖ” the frontend polls for progress updates instead of receiving push notifications

==== API Interface

The `PhysicalAiApi` abstract class defines the contract. Both the frontend proxy and backend implementation conform to it.

.Current API (15 methods)
[source,typescript]
----
export abstract class PhysicalAiApi {
  // Status
  abstract getStatus(): Promise<string>;

  // Image Catalog
  abstract listCatalogImages(namespace: string): Promise<QuayRepository[]>;
  abstract getImageTags(namespace: string, name: string): Promise<QuayTag[]>;
  abstract pullImage(fullImageName: string, tag: string): Promise<void>;
  abstract getPullProgress(image: string): Promise<PullProgress | null>;
  abstract listLocalImages(): Promise<string[]>;

  // Build & Push
  abstract buildBaseImage(tag: string): Promise<void>;
  abstract buildSimulationImage(tag: string, config: SimulationConfig): Promise<void>;
  abstract cancelBuild(tag: string): Promise<void>;
  abstract getBuildProgress(tag: string): Promise<BuildProgress | null>;
  abstract pushImage(tag: string): Promise<void>;
  abstract getPushProgress(tag: string): Promise<PushProgress | null>;

  // Configuration
  abstract getDefaultNamespace(): Promise<string>;
  abstract getSimulationConfig(): Promise<SimulationConfig>;
  abstract saveSimulationConfig(config: SimulationConfig): Promise<void>;
}
----

== Key Technical Decisions

[cols="1,2,2"]
|===
| Decision | Choice | Rationale

| Extension template
| Full template (3-package)
| Extension has substantial UI вҖ” catalog, dashboards, panels

| Frontend framework
| Svelte 5 + TailwindCSS
| Podman Desktop standard; layout via Tailwind, theming via `.pai-*` classes in `app.css` using `--pd-*` CSS variables

| Frontend-backend communication
| MessageProxy RPC with frontend polling
| Podman Desktop webview does not deliver unsolicited backend-to-frontend messages, so frontend polls for progress (every 500ms while active)

| Routing
| tinro (hash mode)
| Required for webview context; state persisted via `PodmanDesktopApi.setState()`

| Build tooling
| Vite 8
| Fast builds, native TypeScript support, matches Podman Desktop ecosystem

| Container base (Jazzy)
| Ubuntu 24.04 (interim)
| ROS2 Jazzy has no official Fedora packages; Ubuntu is the official ROS2 platform. Fedora migration parked until official packaging matures (see APPENG-5769-01)

| Container base (simulation)
| Digest-pinned presets (sloretz multi-arch, osrf amd64-only)
| Presets ensure reproducible builds. sloretz default enables Apple Silicon / Podman on Mac. Pinned digests in `SimulationBaseImages.ts`, bumped deliberately.

| Middleware
| DDS (default); Zenoh planned
| DDS is the ROS2 default and is implemented now. Zenoh with DDS bridge is planned for multi-robot fleet communication (Story 3)

| Build cancellation
| `AbortController` per build tag
